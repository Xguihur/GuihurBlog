## 一条最实用的数据流

可以先把这些概念放进一条数据流里理解：

```text
前端/调用方 -> Controller -> DTO -> Service -> Mapper/DAO -> Entity -> 数据库
数据库 -> Entity -> Service -> VO -> Controller -> 前端/调用方
```

这条链路里，每个对象关注的都不是同一件事。

## DAO 和 Mapper 是什么关系

`DAO` 是更通用的叫法，意思是“数据访问对象”，负责和数据库打交道。

`Mapper` 则更常见于 MyBatis / MyBatis-Plus 体系里，本质上承担的也是数据访问职责。

可以先这样记：

- `DAO`：角色名，强调“这是数据访问层”
- `Mapper`：在 MyBatis 体系里的常见具体命名

很多团队会混着说“去 DAO 层查一下”或者“去 Mapper 里看 SQL”，大多数时候说的是同一层。

## Entity 是什么

`Entity` 通常表示“数据库表在代码中的映射对象”。

它的特点是：

- 最接近数据库表结构
- 一张表通常对应一个 Entity
- Mapper 查出来、保存进去的对象，很多时候就是 Entity

所以可以把它理解成：数据库在业务代码里的“结构化映射”。

## DTO 是什么

`DTO` 是 `Data Transfer Object`，数据传输对象。

它更偏向“输入”，最常见的场景就是接口入参。

DTO 的作用主要有这些：

- 描述接口需要什么参数
- 避免前端直接传整张表对象
- 方便做参数校验
- 隔离接口结构和数据库结构

可以把 DTO 理解成：数据“进入系统时的包裹”。

## VO 是什么

`VO` 常见理解是 `View Object`，偏向“展示对象”。

它更偏向“输出”，用于返回给前端或调用方。

例如一个页面要展示：

- 用户名
- 公司名
- 余额
- 路由名称

这些字段可能来自多张表，这时就不适合直接返回某张表的 Entity，而是更适合组装成一个 VO。

可以把 VO 理解成：数据“离开系统时给外部看的包裹”。

## 它们各自负责什么

最实用的理解方式不是死背英文，而是看职责：

| 概念 | 主要职责 | 常见出现位置 |
| --- | --- | --- |
| `DTO` | 接收入参、跨层传参 | Controller 入参 |
| `Entity` | 映射数据库表结构 | Mapper / 持久化层 |
| `VO` | 返回展示结果 | Controller 出参 |
| `Mapper/DAO` | 执行查询、更新、持久化 | 数据访问层 |

所以一个典型流程通常是：

1. 前端传入 DTO
2. Controller 接收 DTO
3. Service 调用 Mapper 查询或修改 Entity
4. Service 做业务处理
5. Service 组装 VO
6. Controller 返回 VO

## 为什么不直接全用 Entity

这是后端里非常常见的问题。

如果输入、输出、持久化都直接用 Entity，通常会有这些风险：

- 前端可能传入本不该开放的字段
- 数据库结构一改，接口就容易跟着受影响
- 页面展示往往是多表拼装，不是单表结构
- Entity 里可能包含敏感字段，不适合直接返回

所以分出 DTO 和 VO，本质上是在给“输入”和“输出”做隔离。

## 一句话区分

可以先用这版记忆：

- `Entity`：数据库长什么样
- `DTO`：接口收什么
- `VO`：接口回什么
- `Mapper/DAO`：怎么查库、改库

## 快速判断一个类属于哪一类

以后看到一个类，可以先问自己这几个问题：

1. 它是不是直接对应表字段？
2. 它是不是接口参数？
3. 它是不是页面返回结果？

通常对应关系是：

- 对应表字段，大概率是 `Entity`
- 作为接口入参，大概率是 `DTO`
- 作为接口出参，大概率是 `VO`

再结合代码位置去看，会更快：

- Controller 方法参数里，常见 `DTO`
- Mapper / Service 查库时，常见 `Entity`
- 返回结果对象里，常见 `VO`
